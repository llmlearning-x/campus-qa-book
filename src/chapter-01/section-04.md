```
┌─────────────────────────────────────────────────────┐

                       │ HTTP / SSE
┌──────────────────────▼──────────────────────────────┐
│               FastAPI 后端（端口 8000）               │
│  ┌────────────┐  ┌────────────┐  ┌───────────────┐  │
```

## 各组件职责说明

### 前端（Vue3 + Vite）
**它做什么**：用户直接交互的界面层

核心页面：
- `/login` — 登录/注册
- `/admin/knowledge` — 知识库管理（上传文档）
- `/chat` — 问答聊天

技术选型：
- **Vue3**：目前国内使用最广泛的前端框架
- **Element Plus**：Vue3 的 UI 组件库，提供现成的表格、表单、对话框等
- **Pinia**：Vue3 的状态管理（替代 Vuex）
- **Axios**：HTTP 请求库

---

### 后端（Python + FastAPI）
**它做什么**：业务逻辑处理 + 数据协调中枢

FastAPI 是 Python 生态里最现代的 Web 框架，特点：
- 自动生成 Swagger 接口文档
- 基于 Python 类型注解，代码即文档
- 异步支持，性能出色

核心模块：
```
app/
├── main.py              # 入口
├── api/
│   ├── user.py          # 用户相关接口
│   ├── knowledge.py     # 知识库接口
│   └── chat.py          # 问答接口
├── models/              # 数据库模型（SQLAlchemy ORM）
├── schemas/             # 请求/响应数据结构（Pydantic）
├── services/            # 业务逻辑层
│   ├── auth.py          # 认证服务
│   ├── doc_processor.py # 文档处理
│   └── rag.py           # RAG 核心
└── core/
    ├── config.py        # 配置管理
    └── database.py      # 数据库连接
```

---

### 数据库（MySQL）
**它存什么**：结构化数据，包括用户信息、文档元数据、问答记录

关键表：
- `users` — 用户账号信息
- `documents` — 文档元数据（文件名、大小、处理状态）
- `document_chunks` — 切分后的文档片段
- `qa_records` — 历史问答记录

> **为什么用 MySQL 而不是其他数据库？**
> MySQL 是最主流的关系型数据库，适合存储结构化数据。向量数据我们单独用 FAISS 处理，两者各司其职。

---

### 向量库（FAISS）
**它存什么**：文档片段的向量表示，用于语义检索

FAISS（Facebook AI Similarity Search）是 Meta 开源的高性能向量检索库。

在我们的系统里，每个文档片段都有对应的向量。当用户提问时，把问题转成向量，在 FAISS 里找最相近的 K 个片段，这就是检索的核心。

---

### LLM API
**它做什么**：接收 Prompt，生成自然语言回答

我们的系统通过 API 调用大模型，而不是在本地部署。这样做的原因：
- 无需 GPU，普通笔记本就能运行
- 最新的模型能力随时可用
- 按 Token 付费，成本可控

可以使用：
- **OpenAI API**（GPT-4o 等，需要国际网络）
- **阿里云百炼**（通义千问，国内直接访问）
- **讯飞星火**、**文心一言 API** 等

## 数据流全链路

### 离线流程：建立知识库

```
管理员上传 PDF/Word/TXT 文档
  ↓
后端接收文件，保存到服务器
  ↓
文档处理服务：
  1. 提取文本内容
  2. 文本切分（每段 500~1000 字，有重叠）
  3. 批量调用 Embedding API 获取向量
  4. 向量 + 原文 存入 FAISS
  5. 更新 MySQL 文档状态为"已处理"
```

### 在线流程：用户提问

```
用户输入问题："食堂几点开门？"
  ↓
前端发送请求到后端 POST /api/chat
  ↓
后端：
  1. 验证用户登录状态（JWT Token）
  2. 对问题做 Embedding
  3. FAISS 检索 Top5 相关片段
  4. 构建 Prompt（系统指令 + 检索片段 + 用户问题）
  5. 调用 LLM API（流式模式）
  6. 通过 SSE 实时返回给前端
  7. 保存问答记录到 MySQL
  ↓
前端接收流式响应，逐字展示
```

## 本节小结

- 系统由前端、后端、MySQL、FAISS、LLM API 五个核心组件构成
- 离线阶段：文档 → 切分 → 向量化 → 存库
- 在线阶段：提问 → 向量检索 → 拼 Prompt → LLM 生成 → 流式返回
- 各组件职责清晰，可以分模块并行开发
