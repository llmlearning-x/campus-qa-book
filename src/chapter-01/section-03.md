# 1.3 AI 基础认知：LLM 与 RAG 原理

## 从一个问题开始

> "我能不能把我们学校的所有文件上传给 ChatGPT，然后让它回答问题？"

这个想法很自然，但有两个实际问题：
1. **上下文长度限制**：即使是最强的大模型，也有 Token 数量限制（几万到几十万）。一个学校的文档库可能有几百万字，根本塞不进去。
2. **成本问题**：每次对话都发送几百万字的内容，API 费用会高得离谱。

**RAG（检索增强生成）** 就是解决这个问题的方案。在讲 RAG 之前，我们先打好基础概念。

---

## 什么是大模型（LLM）？

LLM（Large Language Model，大语言模型）是一种基于 **Transformer** 架构训练的 AI 模型，通过学习海量文本数据，获得了理解和生成自然语言的能力。

你可以把它想象成一个「读了互联网上所有文章」的超级助手——它知道很多，但有明确的知识截止日期（训练数据的截止时间），也不知道你私有文档里的内容。

**常见的 LLM：**
- GPT-4、GPT-4o（OpenAI）
- Claude（Anthropic）
- 文心一言（百度）
- 通义千问（阿里）
- DeepSeek（深度求索）

对于我们的项目，LLM 的职责是：**给定一段上下文文本和用户的问题，生成一个自然流畅的回答**。

---

## 什么是 Embedding（向量化）？

这是 RAG 系统中最核心的概念之一，也是最难直觉理解的。我们用一个比喻来解释。

### 直觉理解

想象有一个「语义坐标系」。在这个空间里，**语义相近的文本，对应的点也相近**：

```
高维向量空间（简化为2D示意）：

           "猫咪很可爱"
          •
         • "我喜欢小猫"
        •  "宠物店卖猫"
                      ← 这些文本在向量空间里靠得很近

                           "股票大涨"
                          •
                        •  "A股今天涨停"
                        •  "基金净值上升"
```

Embedding 模型就是把一段文字变成这个高维空间里的一个坐标点（一个数字数组）的工具。

### 实际示例

```python
from openai import OpenAI

client = OpenAI()

# 把文本转成向量
response = client.embeddings.create(
    model="text-embedding-3-small",
    input="食堂几点开门？"
)

vector = response.data[0].embedding
print(f"向量维度：{len(vector)}")      # 1536
print(f"前5个数字：{vector[:5]}")      # [-0.012, 0.083, -0.041, ...]
```

一个 1536 维的向量——就是 1536 个浮点数，代表这句话在语义空间里的「坐标」。

## 什么是 RAG？

RAG = **R**etrieval（检索）+ **A**ugmented（增强）+ **G**eneration（生成）

用大白话说：**先从知识库里找到相关内容，再把这些内容塞给大模型，让它基于这些内容来回答。**

### RAG 完整流程

```
【离线阶段 — 建库】
文档 → 切分(Chunking) → 向量化(Embedding) → 存入向量库(FAISS)

【在线阶段 — 问答】
用户提问 → 向量化 → 向量库检索(TopK) → 拼接Prompt → LLM生成 → 返回答案
```

### 具体例子

**文档库里有这段话**（来自学校通知）：
> "中央食堂营业时间：早餐 7:00-9:00，午餐 11:00-13:30，晚餐 17:00-19:30。"

**用户提问**：
> "食堂几点开门？"

**RAG 的处理过程**：
1. 把用户问题向量化：`embed("食堂几点开门？")` → `[0.12, -0.05, ...]`
2. 在向量库里找最相似的文档片段，找到上面那句话
3. 把问题和检索结果拼成 Prompt：
   ```
   你是校园助手，根据以下资料回答问题。
   
   资料：中央食堂营业时间：早餐 7:00-9:00，午餐 11:00-13:30，晚餐 17:00-19:30。
   
   问题：食堂几点开门？
   ```
4. LLM 基于这段资料回答：
   > "中央食堂早餐从 7 点开始，午餐从 11 点开始，晚餐从 17 点开始。"

这就是 RAG 的核心思路：**不是让 AI 凭记忆回答，而是先找证据，再基于证据回答。**

## RAG vs Fine-tuning：选哪个？

| 维度 | RAG | Fine-tuning |
|------|-----|------------|
| 知识更新 | 实时，上传新文档即可 | 需要重新训练，成本高 |
| 实现成本 | 低，1~2 周可落地 | 高，需要大量标注数据和算力 |
| 可解释性 | 高，可以追溯到具体来源文档 | 低，「黑盒」 |
| 准确性 | 依赖检索质量 | 训练好后效果稳定 |
| 适用场景 | 私有文档问答、企业知识库 | 特定风格/领域的能力增强 |

**结论**：对于「文档问答」这类场景，**RAG 是更实用、更快速的选择**。这也是为什么我们选择 RAG 架构来构建校园问答助手。

---

## 本节小结

- LLM 是「读了很多文章」的语言模型，能生成流畅自然的回答，但不了解你的私有数据
- Embedding 把文本变成向量，让语义相近的文本在数学上也「靠近」
- RAG = 先检索相关文档片段，再让 LLM 基于这些片段回答
- 相比 Fine-tuning，RAG 更适合需要实时更新知识的场景
