# 1.2 团队分组与角色分配

## 为什么需要分工？

一个人写代码当然可以，但一个人做系统的上限很低。**分工协作**让每个人能专注在自己擅长的领域，同时并行推进多条工作线。

对于我们的校园问答助手项目，涉及的工作范围包括：
- 前端界面（用户看到的页面）
- 后端 API（数据处理逻辑）
- 数据库设计（数据如何存储）
- AI/算法（向量化、RAG 逻辑）
- 项目管理（协调进度）

---

## 推荐角色配置（4~6 人团队）

### 产品负责人（Product Owner）× 1
**核心职责**：
- 定义功能优先级
- 编写需求文档
- 验收每天的开发成果
- 协调团队沟通

**需要的技能**：沟通能力、逻辑思维、文档写作

> 即使是技术实训，也需要一个人负责「想清楚要做什么」。很多项目失败不是因为技术不行，而是做了没人需要的功能。

---

### 前端工程师（Frontend Engineer）× 1~2
**核心职责**：
- 实现 Vue3 用户界面
- 调用后端 API 展示数据
- 实现聊天交互、文件上传等 UI 功能

**需要的技能**：HTML/CSS 基础、JavaScript、Vue3 框架

**工作成果**：
- 登录/注册页面
- 知识库管理页面
- 聊天问答页面

---

### 后端工程师（Backend Engineer）× 1~2
**核心职责**：
- 实现 FastAPI 接口
- 用户认证、数据库操作
- 连接前端和 AI 模块

**需要的技能**：Python、HTTP 协议、SQL 基础

**工作成果**：
- 用户管理 API
- 知识库管理 API
- 问答接口 `/chat`

---

### AI/算法工程师（AI Engineer）× 1
**核心职责**：
- 实现文档处理（切分 + 向量化）
- 构建 FAISS 向量库
- 实现 RAG 检索逻辑
- 接入 LLM API

**需要的技能**：Python、基础数学（向量、相似度）、了解 LLM API

**工作成果**：
- 文档处理 Pipeline
- 向量检索模块
- RAG 问答模块

## Git 协作基础

多人在同一代码库工作，必须有规范。我们使用以下分支策略：

```
main          ← 只放稳定、可发布的代码
  └── dev     ← 日常开发的集成分支
        ├── feature/user-auth   ← 用户认证功能
        ├── feature/knowledge   ← 知识库功能
        └── feature/chat        ← 问答功能
```

**基本工作流程**：

```bash
# 1. 从 dev 创建自己的功能分支
git checkout -b feature/user-auth

# 2. 开发 + 提交
git add .
git commit -m "feat: 实现用户注册接口"

# 3. 推送到远程
git push origin feature/user-auth

# 4. 在 GitLab 创建 Merge Request，让队友 Review
# 5. 通过 Review 后合并到 dev
```

**Commit 信息规范**（让历史记录更清晰）：

| 前缀 | 含义 | 示例 |
|------|------|------|
| `feat:` | 新功能 | `feat: 添加用户注册接口` |
| `fix:` | Bug 修复 | `fix: 修复登录Token过期问题` |
| `docs:` | 文档更新 | `docs: 更新README安装说明` |
| `refactor:` | 代码重构 | `refactor: 重构文档处理模块` |
| `chore:` | 杂项 | `chore: 更新依赖版本` |

## 本节小结

- 合理分工让团队并行工作，大幅提高效率
- 每个角色都有明确的职责和交付物
- Git 分支策略避免代码冲突，规范 commit 信息便于回溯

## 动手任务

1. 确定团队成员的角色分工，填写角色分配表
2. 每位成员注册 GitLab 账号（下一章会用到）
3. 了解基本的 Git 命令：`clone`、`add`、`commit`、`push`、`pull`
