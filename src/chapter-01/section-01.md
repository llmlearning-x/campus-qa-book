# 1.1 软件开发行业与项目开发流程

## 软件开发是怎么一回事？

如果你是第一次接触软件工程，你可能会觉得「写代码」就是软件开发的全部。但实际上，写代码只占整个开发周期的一小部分。

### 软件开发全流程

软件开发是一个迭代过程：

```
需求分析 → 系统设计 → 编码实现 → 测试验证 → 部署上线 → 维护迭代
       ↑                                            │
       └────────────────────────────────────────────┘
```

> **💡 关键认知**：线上运行后还会收集用户反馈，重新进入需求分析阶段。

一个软件项目的完整生命周期通常包括：

```
需求分析 → 系统设计 → 编码实现 → 测试验证 → 部署上线 → 维护迭代
```

每个阶段都有明确的产出物：

| 阶段 | 产出物 | 参与角色 |
|------|--------|---------|
| 需求分析 | 需求文档（PRD） | 产品经理、客户 |
| 系统设计 | 架构图、数据库设计、接口文档 | 架构师、开发工程师 |
| 编码实现 | 代码、单元测试 | 前端/后端工程师 |
| 测试验证 | 测试报告、Bug 单 | 测试工程师 |
| 部署上线 | 可运行的系统 | 运维工程师 |
| 维护迭代 | 版本更新 | 全体 |

> **💡 关键认知**：「写代码」不等于「软件开发」。一个没有设计的系统，就像没有图纸的建筑——能建起来，但迟早会塌。

## 我们的实训项目：校园知识问答助手

在接下来 6 天里，我们要完整走一遍这个流程，交付一个真实可用的产品：**校园知识问答助手**。

**它能做什么？**

1. 管理员上传学校相关文档（校规、课程信息、活动通知等）
2. 系统自动处理文档，建立知识库
3. 学生用自然语言提问（"食堂几点开门？" "期末考试怎么申请缓考？"）
4. 系统检索知识库，结合 AI 大模型给出准确回答

这不是一个玩具项目。它包含了现代 AI 应用的完整技术栈，也是许多互联网公司正在落地的「企业知识库」产品的核心架构。

## 瀑布模型 vs 敏捷开发

传统的软件开发采用**瀑布模型**：严格按阶段顺序执行，每个阶段完成后才能进入下一阶段。

```
需求 ──► 设计 ──► 开发 ──► 测试 ──► 上线
```

这种方式的问题在于：当你到了测试阶段才发现需求理解有误，修改成本已经很高了。

现代互联网公司大多采用**敏捷开发**：把大功能拆成小迭代（Sprint），每 1~2 周交付一个可用版本，持续收集反馈调整方向。

```
[Sprint 1] 最小可用版本 ──► 反馈 ──► [Sprint 2] 改进版本 ──► 反馈 ──► ...
```

我们的 6 天实训，本质上就是一个压缩版的敏捷开发周期：

- **Day 1**：需求分析 + 技术选型（相当于 Sprint 0）
- **Day 2**：工程搭建（基础设施）
- **Day 3~5**：核心功能迭代开发
- **Day 6**：测试、优化、发布（相当于 Release Sprint）

## 版本控制是一切协作的基础

无论团队大小，版本控制都是软件开发的核心基础设施。

**Git 是什么**：一个分布式版本控制系统，记录代码的每一次变更。

**为什么必须用**：
- 误删代码？`git checkout` 一秒还原
- 多人修改同一文件冲突？`git merge` 帮你整合
- 想知道某个 Bug 是谁引入的？`git blame` 追溯历史
- 想同时开发两个功能不互相干扰？`git branch` 分支隔离

我们用 **GitLab**（企业常用的自部署 Git 服务）来管理代码。具体操作在第2章详细介绍。
